# Почему список заявок мог не обновлять статус (и что сделано)

## Симптом

После смены статуса (назначение, взятие в работу, завершение) **тост внизу** у диспетчера приходит, а **строка в списке** остаётся со старым статусом до ручного F5.

## Причины (сравнение «было / стало»)

| Фактор                           | Было                                                                                                                                                                                                                       | Стало                                                                                                                                        |
| -------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| **Два экземпляра `realtime.ts`** | Turbopack/Next могли загрузить модуль **дважды** → `broadcast()` писал в **пустой** `Set`, а SSE жил в **другом** `Set` → **тосты не приходили**; отмена «обновлялась» только из-за `refreshFullList()` после ответа PATCH | Один глобальный `Set` на **`globalThis.__requestEventsClients`**                                                                             |
| **Тост при своём действии**      | Только SSE                                                                                                                                                                                                                 | Было: JSON **`toast`** + тот же текст по SSE → **дубль**. Стало: тост **только SSE**; дедуп по тексту ~1.5 с (Strict Mode / два EventSource) |
| **Кэш HTTP**                     | Браузер или прокси мог отдавать **старый JSON** для `GET /api/requests`                                                                                                                                                    | Ответ API: `Cache-Control: no-store`; клиент: `cache: 'no-store'` + query `_t` для срыва кэша                                                |
| **Next.js**                      | Route мог кэшироваться как статический                                                                                                                                                                                     | `export const dynamic = 'force-dynamic'` на `GET` заявок                                                                                     |
| **Фильтр диспетчера**            | Один фильтр на весь экран — история строилась из отфильтрованного списка и «пропадала»                                                                                                                                     | Полный снимок **`allRequests`**; фильтр только у карточки **«Все заявки»**; **История** — свой фильтр (завершённые / отменённые / оба)       |
| **SSE → load()**                 | Замыкание на устаревший `load` при смене фильтра / переподписке                                                                                                                                                            | Обновление списка через **ref** (`loadRequestsRef.current()`), всегда актуальная функция                                                     |
| **Мастер**                       | Обновлялся по SSE только при **назначении на себя** + раз в 12 с                                                                                                                                                           | При **любом** `dispatcher_toast` — полная перезагрузка списка мастера (чужие действия тоже отражаются в БД)                                  |

## Ожидаемое поведение сейчас

1. Диспетчер: тост + **обновлённый список** без F5; фильтр при внешнем событии сбрасывается, чтобы заявка не «пропадала» из-за узкого фильтра.
2. Мастер: любое событие по заявкам на сервере → повторный `GET /api/requests` → свои заявки пересчитываются.

## Двойные тосты (исправлено)

1. **Назначение / отмена диспетчером:** после PATCH вызывался `pushToast(j.toast)`, параллельно API делал `broadcast(dispatcher_toast)` с тем же текстом → два тоста. Убран клиентский `pushToast` после успешного assign/cancel.
2. **Мастер взял в работу:** в dev Strict Mode `useEffect` с `EventSource` мог создать **два** подключения → одно SSE-событие обрабатывалось дважды. Добавлена дедупликация: одинаковый текст в течение ~1.5 с не показывается повторно.

## Ограничения

- **Один процесс** (SSE in-memory): несколько реплик без общей шины — тосты не синхронизируются между инстансами.
- Если открыта только вкладка **без** открытого EventSource (редко), до следующего интервала опроса список может отстать на несколько секунд.
